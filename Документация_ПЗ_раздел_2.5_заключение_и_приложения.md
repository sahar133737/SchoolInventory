## 2.5 Тестирование системы

Цель тестирования заключается в обеспечении корректной работы программы для автоматизации учёта школьного имущества в образовательном учреждении.

Тестирование направлено на проверку функциональности графического интерфейса, выявление и устранение возможных ошибок, а также проверку корректности обработки и валидации пользовательского ввода. Основная задача состоит в том, чтобы гарантировать надёжность и устойчивость системы при различных сценариях использования.

Тестирование программного продукта проходило в трёх ситуациях:

- проверка в нормальных условиях;
- проверка в экстремальных условиях;
- проверка в исключительных ситуациях.

Проверку программы в нормальных условиях проведём на примере авторизации в системе, в случае заполнения всех полей корректными данными пользователя. На рисунке 2.5.1 представлено заполнение полей данных в окне авторизации.

### Виды и методы тестирования

В процессе тестирования были применены следующие виды тестирования:

**1) Модульное тестирование (Unit Testing):** проверка корректности работы отдельных классов и методов бизнес-логики, таких как:

- инициализация базы данных и создание структуры БД (DbController: GetConnection, ExecuteNonQuery, GetData, ExecuteScalar);
- валидация вводимых данных (ValidationHelper, DispositionValidator, InputMaskHelper);
- работа с данными через слой доступа (InventoryManager: GetAllInventory, GetFilteredInventory, AddInventoryItem, UpdateInventoryItem, DeleteInventoryItem);
- проверка авторизации пользователей (UserManager: AuthenticateUser, GetUserInfo, InitializeUsersTable);
- формирование отчётов по различным критериям (ReportService: BuildReport);
- учёт распоряжений имуществом (MovementManager: RegisterMovement, GetMovements);
- разграничение прав доступа (PermissionService, AuthorizationHelper);
- экспорт данных в формат, совместимый с Excel (ExcelExporter: ExportToExcelWithDialog);
- экспорт отчётов в PDF (PdfReportExporter: ExportWithDialog);
- ведение журнала событий (Logger: Info, Warning, Error, ReadRecentEntries).

**2) Интеграционное тестирование:** проверка взаимодействия между модулями системы:

- корректность связи Windows Forms интерфейса (DataGridView, ComboBox, TextBox, DateTimePicker, TabControl) с бизнес-логикой и базой данных;
- проверка целостности данных: при добавлении/редактировании инвентаря должны корректно сохраняться все поля в таблице Inventory;
- проверка удаления инвентаря: корректное удаление записи из таблицы Inventory;
- взаимодействие модуля отчётности с модулем данных для формирования выборок по различным критериям (категории, состояние, кабинеты, ответственные, период);
- проверка работы системы авторизации: инициализация таблицы Users, проверка учётных данных, передача информации о пользователе в главную форму;
- проверка модуля «Распоряжение имуществом»: регистрация операции и автоматическое обновление кабинета и ответственного лица в карточке инвентаря;
- проверка экспорта отчётов: корректное формирование CSV-файлов с правильной кодировкой для отображения кириллицы;
- проверка экспорта отчётов в PDF с реквизитами организации;
- проверка работы модуля «Права доступа»: сохранение и применение матрицы прав для ролей Admin и User.

**3) Системное тестирование:** проверка работы приложения в целом на соответствие функциональным требованиям:

- полный сценарий работы пользователя: от авторизации до завершения рабочего сеанса;
- проверка всех основных функций: управление инвентарём (CRUD), распоряжение имуществом, формирование отчётов, статистика, поиск и фильтрация, экспорт данных;
- проверка адаптивности интерфейса при изменении размера окна;
- проверка отображения вкладок в зависимости от роли пользователя (Администратор / Пользователь);
- проверка ленивой загрузки данных при переключении вкладок.

**4) Тестирование пользовательского интерфейса (UI Testing):** проверка корректности отображения элементов интерфейса, реакции на действия пользователя и удобства навигации между разделами:

- проверка всех форм приложения (LoginForm, Form1, InventoryAddDialog, InventoryEditDialog, ReportsForm, StatisticsForm, MovementsForm, UserManagementForm, CategoriesForm, DispositionOperationDialog, RolePermissionsForm, HelpForm);
- проверка корректности отображения текста, отсутствие обрезания кнопок и элементов управления при изменении размера окна;
- проверка работы кнопок, ComboBox, DataGridView, TextBox, DateTimePicker;
- проверка валидации ввода данных и отображения сообщений об ошибках;
- проверка единого светлого стиля интерфейса (AppTheme).

**5) Тестирование конфигурации и развёртывания:** проверка работы приложения на различных компьютерах:

- настройка подключения к БД с различными именами серверов и баз данных (файл App.config);
- работа с Windows Authentication и SQL Server Authentication;
- проверка работы приложения с Microsoft SQL Server (Express, LocalDB, полная версия);
- проверка автоматического создания таблиц Movements, MovementTypes, RolePermissions при первом запуске.

### Сводный финансовый отчёт по категориям

Данный отчёт представляет собой сводку по стоимости инвентаря, сгруппированную по категориям за выбранный промежуток времени. В нём указаны следующие данные: наименование категории, количество позиций, общая стоимость, средняя стоимость единицы, а также итоговые показатели по всем категориям (рисунок В.1). Если при формировании отчёта установлены фильтры по кабинету, состоянию или ответственному лицу, в выборку попадают только соответствующие записи.

Данный отчёт является детализированным перечнем имущества с указанием кабинета размещения и материально ответственного лица. Он представляет собой список позиций инвентаря с указанием инвентарного номера, наименования, категории, кабинета, ответственного, даты постановки на учёт, стоимости и состояния, сгруппированный по кабинетам (рисунок В.2).
