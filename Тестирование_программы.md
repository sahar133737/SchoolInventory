# 8 Тестирование программы

Тестирование разработанного программного обеспечения является критически важным этапом, обеспечивающим соответствие системы установленным функциональным требованиям, надежность работы и удобство использования. Тестирование проводилось на различных этапах разработки и было направлено на проверку всех ключевых модулей Windows Forms приложения «Система инвентаризации VALERA».

## Виды и методы тестирования

В процессе тестирования были применены следующие виды тестирования:

**1) Модульное тестирование (Unit Testing):** проверка корректности работы отдельных классов и методов бизнес-логики, таких как:
   - инициализация базы данных и создание структуры БД (DbController: GetConnection, ExecuteNonQuery, GetData, ExecuteScalar);
   - валидация вводимых данных (например, проверка обязательных полей при добавлении инвентаря) (InventoryAddDialog, InventoryEditDialog);
   - работа с данными через слой доступа (InventoryManager: GetAllInventory, AddInventoryItem, UpdateInventoryItem, DeleteInventoryItem);
   - проверка авторизации пользователей (UserManager: AuthenticateUser, GetUserInfo, InitializeUsersTable);
   - формирование отчетов по различным критериям (ReportsForm: GenerateReport);
   - экспорт данных в Excel формат (ExcelExporter: ExportToExcel, ExportToExcelWithDialog);

**2) Интеграционное тестирование:** проверка взаимодействия между модулями системы:
   - корректность связи Windows Forms интерфейса (DataGridView, ComboBox, TextBox, DateTimePicker, NumericUpDown) с бизнес-логикой и базой данных;
   - проверка целостности данных: при добавлении/редактировании инвентаря должны корректно сохраняться все поля в таблице Inventory;
   - проверка удаления инвентаря: корректное удаление записи из таблицы Inventory;
   - взаимодействие модуля отчетности с модулем данных для формирования выборок по различным критериям (категории, состояние, кабинеты, ответственные);
   - проверка работы системы авторизации: инициализация таблицы Users, проверка учетных данных, передача информации о пользователе в главную форму;
   - проверка экспорта отчетов: корректное формирование CSV файлов с правильной кодировкой для отображения кириллицы;

**3) Системное тестирование:** проверка работы приложения в целом на соответствие функциональным требованиям:
   - полный сценарий работы пользователя: от авторизации до завершения рабочего сеанса;
   - проверка всех основных функций: управление инвентарем (CRUD), формирование отчетов, поиск и фильтрация, экспорт данных;
   - проверка адаптивности интерфейса при изменении размера окна (полноэкранный и оконный режимы);
   - проверка работы неоморфного стиля интерфейса: корректное отображение элементов, эффекты теней, реакции на действия пользователя;

**4) Тестирование пользовательского интерфейса (UI Testing):** проверка корректности отображения элементов интерфейса, реакции на действия пользователя и удобства навигации между разделами:
   - проверка всех форм приложения (LoginForm, Form1, InventoryAddDialog, InventoryEditDialog, ReportsForm);
   - проверка корректности отображения текста, отсутствие "слетания" элементов при изменении размера окна;
   - проверка работы кнопок, ComboBox, DataGridView, TextBox, DateTimePicker, NumericUpDown;
   - проверка валидации ввода данных и отображения сообщений об ошибках;
   - проверка неоморфного стиля: корректное отображение теней, эффектов нажатия кнопок, фокусировки элементов;

**5) Тестирование конфигурации и развертывания:** проверка работы приложения на различных компьютерах:
   - настройка подключения к БД с различными именами серверов и баз данных;
   - работа с Windows Authentication и SQL Server Authentication;
   - сохранение и загрузка настроек подключения из App.config;
   - проверка работы приложения с различными версиями SQL Server (LocalDB, Express, полная версия).

## План тестирования

**Объект тестирования:** Windows Forms приложение «Система инвентаризации VALERA».

**Критерии начала тестирования:**
- завершена разработка всех основных форм приложения (авторизация, главная форма с управлением инвентарем, диалоги добавления/редактирования, форма отчетов);
- реализована функциональность для основных операций (CRUD для инвентаря, формирование отчетов, экспорт в Excel, поиск и фильтрация);
- подготовлена тестовая база данных с набором инвентаря, категорий, кабинетов, ответственных лиц и пользователей;
- реализована адаптивность интерфейса для полноэкранного и оконного режимов;
- реализован неоморфный стиль интерфейса с корректным отображением всех элементов.

**Критерии завершения тестирования:**
- успешно пройдены все запланированные тестовые сценарии;
- исправлены все выявленные критические и серьезные ошибки (блокирующие выполнение основных функций);
- приложение стабильно работает в тестовом окружении;
- подтверждена работоспособность на различных компьютерах с разными настройками БД.

**Окружение для тестирования:**
- Аппаратное обеспечение: Персональный компьютер (ПК).
- Программное обеспечение: ОС Microsoft Windows 10/11, Microsoft Visual Studio 2022, Microsoft SQL Server (LocalDB, Express или полная версия), .NET Framework 4.8.

## Тестовые сценарии и результаты

В ходе тестирования были выполнены ключевые тестовые сценарии, охватывающие основные функции приложения.

### Сценарий 1: Успешная авторизация и навигация по разделам

**Цель:** проверить корректность работы системы безопасности и базовой структуры интерфейса.

**Предусловие:** БД создана и заполнена начальными данными.

**Действия:**
1) запустить приложение. Открывается форма LoginForm;
2) ввести валидные логин и пароль пользователя (например, admin / admin);
3) нажать кнопку "Войти";
4) после успешного входа проверить отображение главной формы Form1 с вкладками: "Инвентарь" и "Отчеты";
5) проверить отображение информации о пользователе в статусной строке;
6) проверить работу кнопок управления инвентарем: "Добавить", "Изменить", "Удалить", "Обновить";
7) переключиться на вкладку "Отчеты" и проверить наличие элементов управления.

**Ожидаемый результат:** система авторизует пользователя, открывает главную форму. Все элементы интерфейса отображаются корректно, кнопки работают, DataGridView загружает данные из БД.

**Фактический результат:** успешно. Авторизация прошла, главная форма отобразилась с информацией о пользователе. Все кнопки меню работают корректно. DataGridView на форме загружает данные из БД. Неоморфный стиль интерфейса отображается корректно.

### Сценарий 2: Добавление нового инвентаря

**Цель:** проверить функциональность модуля управления инвентарем.

**Предусловие:** пользователь авторизован, открыта форма Form1, вкладка "Инвентарь".

**Действия:**
1) нажать кнопку "Добавить";
2) заполнить обязательные поля в форме InventoryAddDialog:
   - Наименование: "Стол офисный";
   - Категория: выбрать из списка (например, "Мебель");
   - Кабинет: выбрать из списка (например, "101 - Кабинет информатики");
   - Ответственный: выбрать сотрудника из списка;
   - Инв. номер: "INV-001";
   - Дата покупки: выбрать дату;
   - Цена: 15000;
   - Состояние: выбрать из списка (например, "Хорошее");
3) заполнить опциональное поле "Описание": "Стол для офисного помещения";
4) нажать кнопку "Сохранить";
5) проверить обновление DataGridView — в списке должен появиться новый инвентарь.

**Ожидаемый результат:** новая запись добавляется в таблицу Inventory. Все поля сохраняются корректно. Инвентарь отображается в DataGridView.

**Фактический результат:** успешно. Инвентарь добавлен. Отображается в списке с корректными данными. При выборе строки в DataGridView данные загружаются в форму для редактирования.

### Сценарий 3: Редактирование существующего инвентаря

**Цель:** проверить функциональность редактирования записей инвентаря.

**Предусловие:** пользователь авторизован, в базе есть записи инвентаря.

**Действия:**
1) в форме Form1 выбрать запись в DataGridView;
2) нажать кнопку "Изменить" или выполнить двойной клик по строке;
3) в открывшейся форме InventoryEditDialog изменить некоторые поля:
   - Наименование: изменить на "Стол офисный (обновленный)";
   - Состояние: изменить на "Требует ремонта";
   - Цена: изменить на 12000;
4) нажать кнопку "Сохранить";
5) проверить обновление DataGridView — изменения должны отобразиться в списке.

**Ожидаемый результат:** запись в таблице Inventory обновляется. Измененные данные корректно отображаются в DataGridView.

**Фактический результат:** успешно. Изменения сохранены. Обновленная информация отображается в списке инвентаря.

**Ожидаемый результат:** запись удаляется из таблицы Inventory. Список обновляется, удаленная запись не отображается.

**Фактический результат:** успешно. Запись удалена. Список обновлен корректно.
