# ПРИЛОЖЕНИЕ А
## Руководство системного программиста

### Назначение программы

Программа "Школьная инвентаризация" предназначена для автоматизации учёта инвентаря в образовательных учреждениях (школах). Она обеспечивает централизованное хранение данных об объектах инвентаря, их местоположении (кабинетах), материально ответственных лицах, категориях и состоянии. Программа позволяет выполнять поиск и фильтрацию записей, формировать отчёты, анализировать статистику и экспортировать данные в Excel.

### Функции программы

**Управление справочниками:**
- ведение списка категорий инвентаря (мебель, компьютерная техника, спортивный инвентарь и т.п.);
- ведение списка кабинетов (местоположений);
- ведение списка ответственных лиц (сотрудников);
- управление пользователями системы.

**Учёт инвентаря:**
- создание, просмотр, редактирование и удаление карточек инвентаря;
- хранение информации о наименовании, описании, категории, кабинете, ответственном лице;
- учёт инвентарного номера, даты покупки, стоимости и текущего состояния.

**Поиск и фильтрация:**
- быстрый поиск по наименованию и описанию;
- фильтрация записей в реальном времени;
- отображение статистики по количеству записей.

**Отчётность и анализ:**
- формирование различных типов отчётов (полный, по категориям, по состоянию, по кабинетам, по ответственным);
- экспорт отчётов в формат CSV (совместимый с Excel);
- статистический анализ данных (общая стоимость, средняя цена, распределение по категориям и состояниям).

**Авторизация и роли:**
- вход пользователей по логину и паролю;
- разграничение прав доступа по ролям (Administrator, User);
- управление пользователями (только для администраторов).

**Работа с базой данных:**
- подключение к существующей базе данных SchoolInventoryDB;
- автоматическая инициализация таблицы пользователей при первом запуске;
- генерация тестовых данных (по 100 записей в каждую таблицу) по запросу администратора.

**Рабочая станция:**
- настольный ПК или ноутбук с ОС Windows 7 или выше.

**Процессор и память:**
- процессор не ниже Intel Core i3 или аналогичный;
- не менее 2 ГБ ОЗУ (рекомендуется 4 ГБ).

### Общая архитектура

Программа реализована как настольное монолитное WinForms-приложение. Архитектура трёхуровневая логически (но физически в одном исполняемом файле):

**Уровень представления (UI):**
- формы LoginForm (авторизация), Form1 (главная форма с вкладками), InventoryAddDialog, InventoryEditDialog (диалоги редактирования), ReportsForm (отчёты), StatisticsForm (статистика), UserManagementForm (управление пользователями), CategoriesForm (управление категориями), HelpForm (справка), TestDataForm (генерация тестовых данных).

**Уровень бизнес-логики:**
- классы для проверки прав доступа, валидации вводимых данных, управления операциями с инвентарём, формирования отчётов и статистики.

**Уровень доступа к данным:**
- класс DbController (работа с SQL Server, выполнение запросов);
- класс InventoryManager (операции с инвентарём);
- класс UserManager (управление пользователями);
- класс CategoryManager (управление категориями);
- модуль Utils.AppConfig (конфигурация приложения);
- модуль Utils.Logger (логирование операций);
- модуль Utils.ValidationHelper (валидация данных);
- модуль Utils.TestDataGenerator (генерация тестовых данных).

Взаимодействие с СУБД осуществляется напрямую через ADO.NET (SqlConnection, SqlCommand), без использования ORM.

**Стилизация:**
- применён неоморфный дизайн (NeomorphicStyle) с тёмной цветовой схемой;
- все элементы интерфейса стилизованы в едином стиле.

### Структура базы данных

База данных SchoolInventoryDB (SQL Server) содержит следующие основные таблицы:

**Categories** — категории инвентаря
- CategoryID (INT, PRIMARY KEY, IDENTITY)
- CategoryName (NVARCHAR)

**Classrooms** — кабинеты (местоположения)
- ClassroomID (INT, PRIMARY KEY, IDENTITY)
- RoomNumber (NVARCHAR)
- RoomName (NVARCHAR)

**ResponsiblePersons** — ответственные лица
- PersonID (INT, PRIMARY KEY, IDENTITY)
- FirstName (NVARCHAR)
- LastName (NVARCHAR)

**Users** — пользователи системы
- UserID (INT, PRIMARY KEY, IDENTITY)
- Username (NVARCHAR, UNIQUE)
- Password (NVARCHAR)
- FullName (NVARCHAR)
- Role (NVARCHAR) — Admin или User
- CreatedDate (DATETIME)
- IsActive (BIT)

**Inventory** — инвентарь
- InventoryID (INT, PRIMARY KEY, IDENTITY)
- ItemName (NVARCHAR) — наименование
- Description (NVARCHAR) — описание
- CategoryID (INT, FOREIGN KEY → Categories)
- ClassroomID (INT, FOREIGN KEY → Classrooms)
- ResponsiblePersonID (INT, FOREIGN KEY → ResponsiblePersons)
- InventoryNumber (NVARCHAR) — инвентарный номер
- PurchaseDate (DATETIME) — дата покупки
- PurchasePrice (DECIMAL) — цена
- CurrentState (NVARCHAR) — состояние (Отличное, Хорошее, Удовлетворительное, Требует ремонта, Неисправное, Списано)
- Status (NVARCHAR) — статус

Для повышения производительности рекомендуется создать индексы на ключевых полях (InventoryNumber, CategoryID, ClassroomID, ResponsiblePersonID, CurrentState).

### Связи с другими программами

Программа работает автономно и не требует обязательного подключения к внешним сервисам. Взаимодействие происходит только с установленным экземпляром SQL Server:
- возможна работа с локальным экземпляром (на той же машине);
- возможна работа с удалённым сервером в локальной сети.

**Экспорт данных:**
- экспорт отчётов и статистики в формат CSV (совместимый с Microsoft Excel);
- файлы сохраняются с кодировкой UTF-8 с BOM для корректного отображения кириллицы.

Обмена с внешними системами (1С, АСУ, облако и др.) в текущей версии не предусмотрено.

### Установка и первоначальный запуск

**1. Установка SQL Server:**
- установить Microsoft SQL Server (Express достаточно) на целевой компьютер или сервер;
- убедиться, что служба SQL Server запущена и известен адрес сервера (например, localhost или SERVER\INSTANCE).

**2. Подготовка базы данных:**
- создать базу данных SchoolInventoryDB на SQL Server;
- создать таблицы согласно структуре (Categories, Classrooms, ResponsiblePersons, Inventory, Users);
- при необходимости заполнить справочники начальными данными.

**3. Установка программы:**
- запустить установку (ClickOnce-публикация или MSI) и следовать инструкциям;
- по умолчанию программа устанавливается в профиль пользователя.

**4. Настройка подключения:**
- отредактировать файл App.config и указать строку подключения к базе данных:
```xml
<connectionStrings>
    <add name="DefaultConnection" 
         connectionString="Server=ИМЯ_СЕРВЕРА;Database=SchoolInventoryDB;Integrated Security=true;" />
</connectionStrings>
```
- или изменить строку подключения в коде DbController.cs (для разработки).

**5. Первый запуск:**
- при первом старте отображается форма LoginForm;
- по умолчанию создаётся пользователь admin/admin (если таблица Users пуста);
- после авторизации открывается главная форма с вкладками.

### Наполнение базы начальными и тестовыми данными

**Начальные данные:**
- таблица Users автоматически создаётся при первом запуске (если отсутствует);
- создаётся пользователь по умолчанию: логин "admin", пароль "admin", роль "Admin".

**Тестовые данные (массовые):**
- по запросу администратора могут быть сгенерированы тестовые данные через форму TestDataForm;
- генерация создаёт по 100 записей в каждую таблицу:
  - 100 пользователей (логины user001-user100, пароль 123456, 10% админов);
  - 100 категорий (включая стандартные: Мебель, Компьютерная техника и т.д.);
  - 100 кабинетов (номера 001-100 с названиями);
  - 100 ответственных лиц (случайные имена и фамилии);
  - 100 записей инвентаря (случайные предметы с реалистичными данными);
- генерация выполняется асинхронно с отображением прогресса;
- дубликаты автоматически пропускаются (по уникальным ключам).

### Настройка параметров приложения

**Параметры подключения к БД:**
- строка подключения хранится в App.config (connectionStrings/DefaultConnection);
- при отсутствии настройки используется значение по умолчанию из DbController.cs;
- может быть изменена администратором через редактирование App.config.

**Имя базы данных:**
- зафиксировано как SchoolInventoryDB (может быть изменено в строке подключения).

**Язык интерфейса:**
- русский; смена языка в текущей версии не поддерживается.

**Визуальное оформление:**
- применён неоморфный дизайн с тёмной цветовой схемой;
- цвета настраиваются в классе NeomorphicStyle.cs;
- основные цвета: BackgroundColor (30,30,40), SurfaceColor (40,40,50), AccentColor (100,150,255).

**Логирование:**
- все операции логируются в папку Logs в директории программы;
- файлы логов создаются ежедневно (log_YYYYMMDD.txt);
- автоматическая очистка логов старше 30 дней (настраивается в App.config: LogRetentionDays);
- уровни логирования: INFO, WARNING, ERROR, DEBUG.

### Контрольные примеры

**Сценарий «Первый запуск и авторизация»:**
1) Запуск программы, отображение формы LoginForm.

Проверка: в каждой таблице создано по 100 записей, данные корректны, программа работает стабильно с большим объёмом данных.

- реализация синхронизации данных между несколькими экземплярами программы.

**Безопасность:**
- реализация хеширования паролей (вместо хранения в открытом виде);
- реализация журналирования действий пользователей;
- реализация резервного копирования базы данных.

### Сообщения при настройке

**«Не удалось подключиться к серверу»:**
- проверить имя сервера в строке подключения (App.config или DbController.cs);
- проверить доступность SQL Server (запущена ли служба);
- проверить корректность учётных данных (для SQL Authentication);
- проверить настройки файрвола (разрешён ли доступ к SQL Server);
- проверить наличие базы данных SchoolInventoryDB на сервере.

**«База данных SchoolInventoryDB не найдена»:**
- создать базу данных вручную на SQL Server;
- или изменить строку подключения на существующую базу данных.

**«Ошибка инициализации таблицы пользователей»:**
- проверить права доступа к базе данных (необходимы права на создание таблиц);
- проверить логи в папке Logs для детальной информации об ошибке.

### Сообщения при выполнении программы

**«Ошибка подключения к базе данных»:**
- проверить, что сервер SQL запущен и доступен;
- проверить строку подключения в App.config;
- проверить права доступа к базе данных SchoolInventoryDB;
- проверить логи в папке Logs для детальной информации.

**«Ошибка выполнения запроса» + детали:**
- проанализировать текст ошибки SQL Server (обычно связана с отсутствием таблицы, несоответствием типов данных или прав доступа);
- проверить логи в папке Logs для полного текста ошибки и стека вызовов;
- убедиться, что все таблицы созданы и имеют правильную структуру.

**«Не удалось загрузить данные инвентаря»:**
- проверить наличие таблицы Inventory в базе данных;
- проверить наличие связанных таблиц (Categories, Classrooms, ResponsiblePersons);
- проверить целостность внешних ключей;
- проверить логи для детальной информации об ошибке.

**«Ошибка генерации данных: Недопустимая операция в нескольких потоках»:**
- ошибка исправлена в версии 2.0, обновить программу до последней версии;
- если ошибка повторяется, проверить логи и сообщить разработчику.

**«Генерация тестовых данных завершена успешно!»:**
- тестовые данные созданы, можно приступать к работе с программой;
- рекомендуется обновить данные на вкладке "Инвентарь" для отображения новых записей.

**«Доступ запрещен» (при попытке доступа к функциям администратора):**
- функция доступна только пользователям с ролью "Admin";
- обратиться к администратору для получения прав доступа или выполнения операции.

**«Не удалось удалить категорию. Возможно, к ней привязаны записи инвентаря»:**
- перед удалением категории необходимо удалить или переместить все записи инвентаря, использующие эту категорию;
- или изменить категорию у связанных записей на другую.

### Логирование и отладка

**Расположение логов:**
- папка Logs в директории программы (рядом с исполняемым файлом);
- файлы создаются ежедневно: log_YYYYMMDD.txt (например, log_20241220.txt);
- автоматическая очистка логов старше 30 дней (настраивается в App.config).

**Уровни логирования:**
- INFO — информационные сообщения (успешные операции, загрузка данных);
- WARNING — предупреждения (некритичные проблемы);
- ERROR — ошибки (исключения с полным стеком вызовов);
- DEBUG — отладочная информация (только в Debug-версии).

**Использование логов для диагностики:**
- при возникновении ошибок проверять последние записи в лог-файле;
- искать записи с уровнем ERROR для выявления проблем;
- использовать временные метки для отслеживания последовательности событий.
